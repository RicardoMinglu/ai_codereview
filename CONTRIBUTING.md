# 贡献指南

感谢你愿意改进本项目。以下为约定流程，便于评审与合并。

## 基本原则

- **默认分支**：向 **`master`** 提交 PR（见 README）。
- **范围**：改动与 PR 描述一致；避免无关格式化或「顺手重构」。
- **测试**：请运行并通过：
  ```bash
  go vet ./...
  go test ./... -count=1
  ```
- **配置与密钥**：勿将真实 `config.yaml`、Token、Webhook 密钥等提交入库；仅更新 `config.example.yaml` 等模板时使用占位符。

## 提交 PR

1. Fork 本仓库，从 **`master`** 拉出功能分支（命名建议：`feat/xxx`、`fix/xxx`）。
2. 提交信息建议简短说明动机（可用中文或英文）。
3. 发起 PR 时请注明：
   - **动机 / 背景**
   - **主要改动**
   - **如何验证**（命令、手工步骤等）

涉及行为变更或新依赖时，建议在 PR 正文或 `docs/` 中补充简要说明。

## 代码风格

- 与现有 Go 代码保持一致的格式与命名；提交前可对改动文件执行 `gofmt`。
- 新增逻辑尽量带表意清晰的测试（尤其是边界与错误路径）。

## 行为准则

请保持尊重、就事论事。对设计有分歧时，优先在 Issue/PR 里写清约束与取舍，便于维护者与其他贡献者跟进。

如有疑问，可先开 Issue 讨论再投入大量实现。
