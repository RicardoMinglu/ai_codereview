# AI Code Review 使用文档

## 项目简介

AI Code Review 是一个基于大语言模型的自动化代码评审服务。当 GitHub 仓库发生 `push` 或 `pull_request` 事件时，系统会自动获取代码变更、调用 AI 进行评审，并生成结构化报告与通知。

### 核心功能

- **自动评审**：监听 GitHub Webhook，按 push/PR 触发评审
- **结构化结果**：生成 0-100 分、问题列表、改进建议
- **多模型支持**：Claude、OpenAI、Gemini，支持 OpenAI 兼容 API
- **存储**：运行时使用 **MySQL**（报告与仓库配置表同库）
- **按仓库配置**：各 `owner/repo` 在 `github_projects` 中一行，独立 Token、评审片段与通知等
- **多通知渠道**：钉钉、企业微信、自定义 Webhook（Slack、飞书等）
- **可视化报告**：Web 页面查看评审列表、详情、仓库筛选

## 快速开始

### 1. 环境要求

- Go 1.25+（源码运行）
- GitHub Personal Access Token（需 `repo` 权限）
- 至少一个 AI API Key（Claude/OpenAI/Gemini）
- **MySQL**（部署前执行 `docs/mysql/init.sql`）

### 2. 安装部署

#### 方式一：源码编译

```bash
# 克隆仓库
git clone https://github.com/RicardoMinglu/ai_codereview.git
cd ai_codereview

# 下载依赖
make deps

# 编译
make build
```

#### 方式二：Docker 部署

```bash
# 构建镜像
make docker-build

# 运行容器
make docker-run
```

### 3. 配置文件

```bash
# 从示例创建配置文件
make init-config

# 编辑配置
vim config.yaml
```

#### 最小配置示例

```yaml
github:
  token: "ghp_xxxxxxxxxxxxxxxxxxxx"

ai:
  provider: "claude"
  claude:
    api_key: "sk-ant-xxxxxxxxxxxxxxxxxxxx"
```

### 4. 启动服务

```bash
# 使用默认配置
make run

# 或指定配置文件
./ai-code-review -config config.yaml
```

### 5. 验证运行

```bash
curl http://localhost:8078/health
# 返回: ok
```

## 详细配置说明

### 服务器配置

```yaml
server:
  port: 8078  # HTTP 端口
  base_url: "http://localhost:8078"  # 对外访问地址，用于生成报告链接
```

### GitHub 配置

```yaml
github:
  token: "ghp_xxxxxxxxxxxxxxxxxxxx"  # 必填，需 repo 权限
  webhook_secret: "your-webhook-secret"  # 可选，生产环境强烈建议配置
```

**获取 GitHub Token：**
1. GitHub → Settings → Developer settings → Personal access tokens
2. Generate new token (classic)
3. 勾选 `repo` 权限
4. 生成并复制 token

### AI 模型配置

#### Claude（推荐）

```yaml
ai:
  provider: "claude"
  claude:
    api_key: "sk-ant-xxxxxxxxxxxxxxxxxxxx"
    model: "claude-sonnet-4-6"  # 可选，默认值
```

#### OpenAI

```yaml
ai:
  provider: "openai"
  openai:
    api_key: "sk-xxxxxxxxxxxxxxxxxxxx"
    model: "gpt-4o"
    # base_url: "https://api.xxx.com/v1"  # 可选，用于中转站
```

#### Gemini

```yaml
ai:
  provider: "gemini"
  gemini:
    api_key: "AIzaSyxxxxxxxxxxxxxxxxxxxxxxxx"
    model: "gemini-2.5-flash"
```

#### 使用 OpenAI 兼容 API（中转站）

```yaml
ai:
  provider: "openai"
  openai:
    api_key: "你的中转站密钥"
    model: "gpt-4o"
    base_url: "https://api.你的中转站.com/v1"
```

### 评审配置

```yaml
review:
  max_diff_lines: 5000  # 最大 diff 行数，超出则截断
  language: "zh"  # 评审输出语言：zh（中文）| en（英文）
  ignore_patterns:  # 忽略的文件模式
    - "*.lock"
    - "*.sum"
    - "vendor/*"
    - "node_modules/*"
    - "*.min.js"
    - "*.min.css"
```

### 数据库配置（仅 MySQL）

```yaml
storage:
  type: "mysql"
  dsn: "user:pass@tcp(127.0.0.1:3306)/ai_review?charset=utf8mb4"
```

**注意：**
- 须先创建数据库，并执行 **`docs/mysql/init.sql`** 建表（服务不会自动建表）
- 仓库级 Token、Webhook、评审与通知见「仓库配置（github_projects）」

### 通知配置

#### 钉钉

```yaml
notify:
  dingtalk:
    enabled: true
    webhook_url: "https://oapi.dingtalk.com/robot/send?access_token=xxx"
    secret: "SECxxxxxxxx"  # 加签密钥
```

#### 企业微信

```yaml
notify:
  wecom:
    enabled: true
    webhook_url: "https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=xxx"
```

#### 第三方 Webhook（Slack、飞书等）

```yaml
notify:
  webhooks:
    - enabled: true
      name: "slack"
      url: "https://hooks.slack.com/services/xxx"
      type: "custom"
```

## GitHub Webhook 配置

### 1. 创建 Webhook

1. 打开 GitHub 仓库 → **Settings** → **Webhooks** → **Add webhook**
2. 填写配置：
   - **Payload URL**: `https://<你的域名>:8078/webhook/github`
   - **Content type**: `application/json`
   - **Secret**: 与 `config.yaml` 中 `github.webhook_secret` 一致
   - **Which events**: 选择 `Pushes` 和 `Pull requests`
3. 点击 **Add webhook**

### 2. 本地调试

使用 ngrok 暴露本地端口：

```bash
# 安装 ngrok
brew install ngrok  # macOS
# 或从 https://ngrok.com/ 下载

# 暴露端口
ngrok http 8078

# 将生成的 https 地址填入 Webhook Payload URL
# 例如：https://abc123.ngrok.io/webhook/github
```

## 仓库配置（github_projects）

每个 GitHub 仓库在表 **`github_projects`** 中占一行；`config.yaml` 中不包含各仓库存取 Token 或通知。新增仓库即新增一行，与团队规模无关。

### 配置方式

#### 方式一：Web 可视化界面（推荐）

访问 `http://localhost:8078/admin/projects` 进入项目管理界面。

**功能：**
- 查看所有仓库配置列表
- 添加 / 编辑 / 删除行
- 启用/禁用某仓库
- 在线 JSON 编辑器（支持语法高亮和验证）

**操作步骤：**
1. 点击「添加项目」
2. 填写仓库全名（`owner/repo`）
3. 填写该行仓库的 **GitHub Token**（必填）
4. 按需填写 **Webhook Secret**（生产建议填写以启用验签）
5. 按需填写 **评审 / 通知 JSON**
6. 保存

#### 方式二：SQL 命令行

在 MySQL 数据库中操作 `github_projects` 表：

```sql
-- 示例 1：仅登记 Token；评审与内置默认合并；不发通知
INSERT INTO github_projects (
  repo_full_name, 
  enabled, 
  github_token, 
  webhook_secret, 
  review_json, 
  notify_json
) VALUES (
  'org/repo-a', 
  1, 
  'ghp_xxx',
  'whsec_xxx',  -- 可 NULL：不验签（仅建议内网/调试）
  '{"max_diff_lines":3000,"language":"zh"}',
  NULL          -- NULL：不发送通知
);

-- 示例 2：Token/Secret + 评审 + 通知均自定义
INSERT INTO github_projects (
  repo_full_name, 
  enabled, 
  github_token, 
  webhook_secret, 
  review_json, 
  notify_json
) VALUES (
  'org/repo-b', 
  1, 
  'ghp_xxx',  -- 独立的 GitHub Token
  'whsec_xxx',  -- 独立的 Webhook Secret
  '{"max_diff_lines":8000,"language":"en","ignore_patterns":["*.test.js"]}',
  '{"dingtalk":{"enabled":true,"webhook_url":"https://oapi.dingtalk.com/robot/send?access_token=xxx","secret":"SECxxx"}}'
);

-- 示例 3：禁用某个仓库
UPDATE github_projects SET enabled = 0 WHERE repo_full_name = 'org/repo-c';
```

### 行为说明

- Webhook **仅处理**已在表中登记且 **`enabled = 1`** 的仓库；未登记仓库的事件会被忽略。
- **`github_token`** 必填（非空），用于调用 GitHub API 拉取变更。
- **`webhook_secret`** 非空时校验 `X-Hub-Signature-256`；为空则不验签。
- **`review_json`** 可选，与内置默认评审配置 **合并**；为空则全部用默认。
- **`notify_json`** 可选；为空则**不发送**通知；非空则为完整通知配置（JSON）。

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `repo_full_name` | string | 仓库全名，格式：`owner/repo` |
| `enabled` | boolean | 是否启用，0=禁用，1=启用 |
| `github_token` | string | **必填**。该仓库使用的 GitHub Token（`repo`） |
| `webhook_secret` | string | 可选。与 GitHub Webhook 密钥一致时启用验签 |
| `review_json` | json | 可选。评审参数片段，与默认配置合并 |
| `notify_json` | json | 可选。完整通知配置；空表示不通知 |

### review_json 配置示例

`review_json` 支持覆盖以下字段（与 `config.yaml` 中 `review` 段一致）：

```json
{
  "max_diff_lines": 8000,
  "language": "en",
  "ignore_patterns": [
    "*.lock",
    "*.sum",
    "vendor/*",
    "test/*"
  ]
}
```

### notify_json 配置示例

`notify_json` 需提供完整的通知配置（与 `config.yaml` 中 `notify` 段一致）：

```json
{
  "dingtalk": {
    "enabled": true,
    "webhook_url": "https://oapi.dingtalk.com/robot/send?access_token=xxx",
    "secret": "SECxxxxxxxx"
  },
  "wecom": {
    "enabled": false
  },
  "webhooks": [
    {
      "enabled": true,
      "name": "slack",
      "url": "https://hooks.slack.com/services/xxx",
      "type": "custom"
    }
  ]
}
```

### 使用场景

#### 场景 1：多个团队共用一个服务
```sql
-- 团队 A 的仓库，使用钉钉通知
INSERT INTO github_projects (repo_full_name, enabled, review_json, notify_json) 
VALUES ('team-a/project1', 1, NULL, 
  '{"dingtalk":{"enabled":true,"webhook_url":"https://oapi.dingtalk.com/robot/send?access_token=team_a_token"}}');

-- 团队 B 的仓库，使用企业微信通知
INSERT INTO github_projects (repo_full_name, enabled, review_json, notify_json) 
VALUES ('team-b/project1', 1, NULL, 
  '{"wecom":{"enabled":true,"webhook_url":"https://qyapi.weixin.qq.com/cgi-bin/webhook/send?key=team_b_key"}}');
```

#### 场景 2：不同仓库使用不同的评审标准
```sql
-- 核心项目，严格评审
INSERT INTO github_projects (repo_full_name, enabled, review_json) 
VALUES ('org/core-service', 1, '{"max_diff_lines":3000,"language":"zh"}');

-- 文档项目，宽松评审
INSERT INTO github_projects (repo_full_name, enabled, review_json) 
VALUES ('org/docs', 1, '{"max_diff_lines":10000,"language":"zh","ignore_patterns":["*.md"]}');
```

#### 场景 3：使用不同的 GitHub Token
```sql
-- 私有仓库，使用专用 Token
INSERT INTO github_projects (repo_full_name, enabled, github_token) 
VALUES ('private-org/secret-repo', 1, 'ghp_private_token_xxx');
```

### 管理命令

```sql
-- 查看所有仓库配置摘要
SELECT repo_full_name, enabled,
       CASE WHEN github_token IS NULL OR github_token = '' THEN '缺 token' ELSE '已填' END AS token_state,
       CASE WHEN review_json IS NULL OR review_json = '' THEN '默认' ELSE '自定义' END AS review_state
FROM github_projects;

-- 启用/禁用
UPDATE github_projects SET enabled = 1 WHERE repo_full_name = 'org/repo';
UPDATE github_projects SET enabled = 0 WHERE repo_full_name = 'org/repo';

-- 删除一行
DELETE FROM github_projects WHERE repo_full_name = 'org/repo';

-- 清空表（无仓库再接收评审，需重新登记）
TRUNCATE TABLE github_projects;
```

### 注意事项

1. **未登记或未启用**：未出现在表中或 `enabled=0` 的仓库不会进入评审。
2. **JSON 格式**：`review_json`、`notify_json` 须为有效 JSON。
3. **立即生效**：表数据变更后无需重启即可作用于新事件。

## 评审维度说明

AI 会从以下维度评审代码：

1. **Bugs（错误）**：逻辑错误、边界情况、空值处理、竞态条件
2. **Security（安全）**：SQL 注入、XSS、硬编码密钥、不安全操作
3. **Performance（性能）**：N+1 查询、不必要的内存分配、阻塞操作
4. **Code Quality（代码质量）**：可读性、命名、重复代码、复杂度
5. **Best Practices（最佳实践）**：错误处理、日志记录、测试、文档

### 问题严重程度

- **Critical（严重）**：必须修复的问题，如安全漏洞、严重 bug
- **Warning（警告）**：应该修复的问题，如性能问题、代码异味
- **Info（信息）**：可以改进的地方，如代码风格
- **Suggestion（建议）**：优化建议，如更好的实现方式

## 常见问题

### Q: 评审没有触发？

**排查步骤：**
1. 检查 Webhook 配置是否正确
2. 查看服务日志是否收到事件
3. 确认 webhook_secret 与 GitHub 配置一致
4. 检查该仓库是否在 `github_projects` 中登记且已启用（`enabled=1`）

### Q: 报告链接打不开？

**解决方案：**
- 确保 `server.base_url` 配置为实际可访问的地址
- 本地调试时使用 ngrok 地址
- 生产环境使用公网域名

### Q: Diff 太大被截断？

**解决方案：**
- 调整 `review.max_diff_lines`（注意 AI 上下文限制）
- 通过 `review.ignore_patterns` 排除无关文件
- 将大型重构拆分为多个小 PR

### Q: 如何切换 AI 模型？

修改 `config.yaml` 中 `ai.provider` 和对应的 `api_key`，重启服务即可。

### Q: 数据存储在哪里？

- **MySQL**：报告与 `github_projects` 均在配置的库中；Docker 部署请挂载数据卷并正确设置 `storage.dsn`

### Q: 如何为不同仓库设置不同的配置？

在 `github_projects` 中为每个 `owner/repo` 维护一行（Web **`/admin/projects`** 或 SQL）。详见上文「仓库配置」。

### Q: 如何查看日志？

服务日志输出到 stdout/stderr，可以重定向到文件：

```bash
./ai-code-review -config config.yaml > app.log 2>&1
```

或使用 systemd/supervisor 管理日志。

## 生产部署建议

### 1. 使用反向代理

使用 Nginx 提供 HTTPS 和负载均衡：

```nginx
server {
    listen 443 ssl;
    server_name code-review.example.com;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    location / {
        proxy_pass http://localhost:8078;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

### 2. 使用进程管理

使用 systemd 管理服务：

```ini
# /etc/systemd/system/ai-code-review.service
[Unit]
Description=AI Code Review Service
After=network.target

[Service]
Type=simple
User=www-data
WorkingDirectory=/opt/ai-code-review
ExecStart=/opt/ai-code-review/ai-code-review -config /opt/ai-code-review/config.yaml
Restart=on-failure

[Install]
WantedBy=multi-user.target
```

启动服务：

```bash
sudo systemctl enable ai-code-review
sudo systemctl start ai-code-review
sudo systemctl status ai-code-review
```

### 3. 监控和告警

- 定期检查 `/health` 端点
- 监控服务日志
- 监控 MySQL 磁盘与连接数

### 4. 安全建议

- **务必配置** `webhook_secret`
- 不要将 API Key 提交到代码仓库
- 使用环境变量或密钥管理服务
- 限制 Web 界面访问（内网/VPN/鉴权）
- 定期更新依赖和 Go 版本

### 5. 性能优化

- 调整 `review.max_diff_lines` 避免超大 diff
- 使用 `review.ignore_patterns` 排除无关文件
- 考虑使用 Redis 缓存评审结果
- 数据库定期清理历史记录
